## 1. How to Stand Out from Other Candidates

### ❌ What Most Candidates Do Wrong

- Spend months on tutorials without building real projects ("tutorial hell")
- Copy-paste code from YouTube without understanding it
- Have GitHub with 0 commits, no README, no live deployments
- Apply to 5 jobs total and wait for responses
- Learn DSA but never practice on a timer or under pressure
- Submit the same generic resume to every job
- Never contact recruiters or build a network
- Have projects with names like "todo-app-v3-final-FINAL"
- Focus only on certification courses, not practical skills
- Never do mock interviews — first real interview is their first practice

### ✅ What Top 1% Candidates Do Differently

- **Build in public** — post progress on LinkedIn weekly, even failures
- **Ship complete products** — not just features, but deployed apps with users
- **Quantity + quality in DSA** — solve 300–400 problems with intent, not just volume
- **Reach out directly** to engineers at target companies on LinkedIn
- **Have a story** — every project has a problem, solution, and measurable impact
- **Treat job hunting as a product** — track applications like a sales funnel (applied → response → interview → offer)
- **Prepare system design early**, even as a fresher — it sets you apart
- **Have referrals** — 60%+ of product company hires come through referrals

### 🏗️ How to Build a Strong Profile

#### GitHub
- [ ] Minimum 5 repositories, each with a detailed README
- [ ] Green contribution graph (commit daily, even small changes)
- [ ] Pin your top 3 projects on your GitHub profile
- [ ] Each repo should have: description, tech stack, live link, screenshots

#### Portfolio Website
- [ ] yourname.dev or yourname.vercel.app
- [ ] Projects section with screenshots and live links
- [ ] About section with your story and skills
- [ ] Directly link to GitHub repos and deployed apps

#### LinkedIn
- [ ] Professional photo, strong headline: "Full-Stack Dev | React + Node.js | Open to Opportunities"
- [ ] 500+ connections (connect with engineers, HRs, fellow students)
- [ ] Post 2x/week: project updates, LeetCode streaks, learning milestones
- [ ] Ask for endorsements from college seniors, professors, internship mentors

### 🔥 How to Show Proof of Skills

- Deploy every project — no excuses. Vercel (frontend), Render/Railway (backend), MongoDB Atlas (DB)
- Add a **real feature** that required non-trivial thinking: real-time chat, payment gateway, OAuth
- Track and show numbers: "500+ users", "reduced load time by 40%", "processes 1000 API calls/day"
- Record a **2-minute Loom video demo** of your project and link it in your README and resume
- Contribute to 1–2 open source projects — even documentation fixes count
- Write **1 blog post per project** on Dev.to or Hashnode explaining what you built and what you learned

### 📊 DSA Roadmap (Beginner → Advanced)

> **Philosophy:** Understand the pattern, not just the solution. 1 well-understood problem > 5 copy-pasted solutions.

#### Phase 1 — Foundations (Weeks 1–4)
**Goal:** Build problem-solving muscle and pattern recognition.

| Topic | Problems to Solve | Focus |
|---|---|---|
| Arrays & Strings | 25–30 | Two pointers, sliding window, prefix sum |
| Hashing (HashMap/Set) | 15–20 | Frequency count, anagram, duplicate detection |
| Basic Math & Bit Manipulation | 10–15 | Modulo, GCD, power of 2 |
| Recursion Basics | 15–20 | Factorial, Fibonacci, subsets |

**Checklist:**
- [ ] Solve every problem without looking at the solution for at least 30 minutes
- [ ] Write your approach in plain English before coding
- [ ] After solving, read 2–3 other solutions to learn patterns

#### Phase 2 — Core Data Structures (Weeks 5–10)
| Topic | Problems to Solve | Focus |
|---|---|---|
| Linked Lists | 20–25 | Reversal, cycle detection, merge |
| Stacks & Queues | 20–25 | Monotonic stack, next greater element |
| Binary Search | 20–25 | Classic, on answers, rotated arrays |
| Trees (Binary Trees + BST) | 30–35 | Traversals, LCA, height, diameter |
| Heaps/Priority Queue | 15–20 | Top-K, merge K sorted lists |

#### Phase 3 — Advanced Topics (Weeks 11–18)
| Topic | Problems to Solve | Focus |
|---|---|---|
| Graphs (BFS/DFS) | 25–30 | Connected components, shortest path |
| Dynamic Programming | 40–50 | 1D, 2D, knapsack, LCS, subsequences |
| Backtracking | 15–20 | Subsets, permutations, N-Queens |
| Tries | 10–12 | Insert, search, prefix matching |
| Sliding Window (Advanced) | 15–20 | Variable window, multi-condition |
| Greedy Algorithms | 15–20 | Intervals, activity selection |

#### 📈 Total Problems to Solve
- **Minimum:** 300 problems
- **Target:** 400 problems
- **Distribution:** ~60% Easy, ~30% Medium, ~10% Hard
- **Platform:** LeetCode (primary), Codeforces (optional for competitive)

#### 🧠 How to Actually Improve Problem-Solving

1. **The 25-Minute Rule:** Attempt every problem for at least 25 minutes independently. If stuck, look at hints only — not full solutions.
2. **Pattern Journal:** Maintain a Notion/Google doc with problem patterns. When you solve a sliding window problem, write: "Trigger: subarray/substring with constraint → use sliding window."
3. **Spaced Repetition:** Re-solve problems you got wrong after 3 days, then 7 days, then 14 days.
4. **Timed Practice:** After Week 6, solve all problems within 25–30 minutes (simulating real interview pressure).
5. **Blind 75 First:** Start with the [Blind 75 list](https://www.teamblind.com/post/New-Year-Gift---Curated-List-of-Top-75-LeetCode-Questions-to-Save-Your-Time-OaM1orEU) — these are the most frequently asked patterns.
6. **Think aloud:** Even when solving alone, verbalize your approach. This builds the habit for interviews.

### 💻 Development Roadmap

#### Frontend — React

**Level Required for Product Companies:** Mid-level React developer who can build scalable, maintainable UIs.

**Topics to Cover:**
- [ ] React fundamentals: JSX, props, state, component lifecycle
- [ ] Hooks: useState, useEffect, useContext, useReducer, useMemo, useCallback
- [ ] React Router v6 (protected routes, nested routes, lazy loading)
- [ ] State management: Context API (must), Redux Toolkit (good to have), Zustand (optional)
- [ ] API integration: Axios/Fetch, loading states, error handling, custom hooks
- [ ] Forms: React Hook Form + Zod validation
- [ ] UI Libraries: Tailwind CSS (required), ShadCN/Radix UI (bonus)
- [ ] Performance: Code splitting, lazy loading, memoization
- [ ] Testing: React Testing Library basics (a plus, not mandatory for freshers)

**What you do NOT need (yet):** Next.js (learn after basics, but add to Phase 3 if time permits)

#### Backend — Node.js + Express

- [ ] Node.js core: event loop, modules, async/await, streams
- [ ] Express.js: routing, middleware, error handling, CORS
- [ ] RESTful API design: proper HTTP methods, status codes, versioning
- [ ] Input validation: express-validator or Joi
- [ ] Authentication: JWT (access + refresh tokens), bcrypt for password hashing
- [ ] OAuth 2.0 (Google Sign-In) — great for projects
- [ ] File uploads: Multer + Cloudinary
- [ ] Rate limiting, helmet.js for security basics
- [ ] WebSockets: Socket.io for real-time features

#### Database

**MongoDB (NoSQL):**
- [ ] Mongoose ODM, schemas, models
- [ ] CRUD operations, aggregation pipeline
- [ ] Indexing, relationships ($lookup, populate)
- [ ] MongoDB Atlas (cloud, free tier)

**PostgreSQL (SQL — must learn basics):**
- [ ] SQL: SELECT, JOIN, WHERE, GROUP BY, subqueries
- [ ] Prisma ORM (modern, TypeScript-friendly)
- [ ] Basic indexing and query optimization
- [ ] Supabase (hosted Postgres, free tier)

> **Rule:** Know one deeply (MongoDB), know the other well enough to use (PostgreSQL). Product companies care that you understand both paradigms.

#### Authentication

- [ ] JWT-based auth flow (login → token → protected routes)
- [ ] Refresh token rotation
- [ ] Google OAuth with Passport.js or next-auth
- [ ] Role-based access control (RBAC) — admin vs user

#### Deployment (Non-Negotiable)

| Service | Use For | Cost |
|---|---|---|
| Vercel | React frontend | Free |
| Render / Railway | Node.js backend | Free tier |
| MongoDB Atlas | Database | Free 512MB |
| Supabase | PostgreSQL | Free tier |
| Cloudinary | Image/file storage | Free tier |
| GitHub Actions | CI/CD basics | Free |

- [ ] Every project MUST be deployed and have a live URL
- [ ] Learn to set environment variables in deployment dashboards
- [ ] Basic Docker understanding (a big plus for senior roles)

> **Rule of thumb:** 1 deployed, well-documented project with real features > 5 half-finished tutorial clones.

**Why:** Every interview will ask about auth. Building it yourself proves you understand security.

**Tech:** React + Node.js + MongoDB + JWT + Tailwind

**Must-Have Features:**
- [ ] Register, Login, Logout with JWT (access + refresh tokens)
- [ ] Email verification via Nodemailer
- [ ] Password reset via email link
- [ ] Google OAuth login
- [ ] Protected routes on frontend
- [ ] Role-based access: admin dashboard vs user dashboard
- [ ] Rate limiting on login (prevent brute force)
- [ ] Proper error handling and form validation

**What Makes It Resume-Worthy:**
- Implemented refresh token rotation with secure httpOnly cookies
- Added Google OAuth reducing registration friction
- Built role-based access control for admin vs regular users

**What Makes It Resume-Worthy:**
- Implemented real-time collaboration using Socket.io, supporting 20+ concurrent users

### How to Balance DSA and Development

- **Never skip DSA to do only development.** DSA is your gate to interviews.

- In Phase 3–4, DSA > development (60/40 split) — polish skills for interviews
- Track consistency with a simple habit tracker (Notion, spreadsheet, or paper)

## 8. Interview Preparation Strategy

### Step-by-Step Interview Preparation Plan

#### 6 Weeks Before Interview Season
- [ ] Complete DSA topics (at least 250+ problems solved)
- [ ] Solve 5 problems per day: 3 medium, 1 easy, 1 hard
- [ ] Start system design basics: read Grokking the System Design Interview summaries
- [ ] Prepare your project story for each project (Problem → Solution → Impact → What you learned)

#### 2–4 Weeks Before Interview
- [ ] Do 1 full mock interview per day (use Pramp or pair up with a friend)
- [ ] Revise: Arrays, Trees, DP, Graphs — the most common interview topics
- [ ] Prepare 10 behavioral answers in STAR format (Situation, Task, Action, Result)
- [ ] Time yourself: solve medium problems in under 20 minutes

#### Week Before Interview
- [ ] No new topics — only revision and mock interviews
- [ ] Read about the company: products, engineering blog, recent news
- [ ] Prepare 5 thoughtful questions to ask the interviewer
- [ ] Sleep well, exercise — physical state affects performance

### How Coding Rounds Work

**Online Assessment (OA):**
- Usually 2–3 problems in 60–90 minutes
- Automated grading — partial credit for passing test cases
- **Strategy:** Read all problems first. Start with the easiest. Aim to pass 60–70% of test cases before perfecting edge cases.

**Technical Interview (Live Coding):**
- 45–60 minutes, 1–2 problems
- Interviewer is watching your process, not just the final code

### How to Communicate During the Interview

**Structure your answer every time:**
1. **Clarify** (2 min): "Can I ask a few clarifying questions? What are the constraints on the input? Should I handle null/empty inputs?"
2. **Plan aloud** (2–3 min): "My first thought is to use a HashMap here because... Let me think about the time complexity..."
3. **Code** (15–20 min): Code while talking. Narrate what you're doing.
4. **Test** (3–5 min): Walk through with a simple example, then an edge case.
5. **Optimize** (if time): "The current solution is O(n²). I think we can improve it to O(n log n) using..."

**Phrases that impress:**
- "Let me think through the brute force first, then we can optimize."
- "The time complexity is O(n) and space is O(1) because..."
- "I want to handle this edge case: what if the input is empty?"

### How to Handle Being Stuck

- Don't go silent. Say: "I know I need to find X. Let me think about what data structure gives me that efficiently..."
- Ask for a hint: "Could I get a small nudge in the right direction?" — interviewers usually appreciate this
- Start with brute force: "Let me start with a solution that works, even if not optimal."
- Think aloud about trade-offs: "I could use recursion here, but it might hit stack overflow for large inputs — a BFS/iterative approach would be safer."

### Mock Interview Strategy

- **Pramp:** Free, peer-to-peer mock interviews (highly recommended)
- **interviewing.io:** Anonymous mock interviews with engineers from top companies (paid)
- **LeetCode Interview Simulator:** Timed, company-specific
- **Peer Practice:** Find a study partner and mock-interview each other 3x/week
- After every mock: write down what went well and what to improve

## 9. Common Mistakes (CRITICAL)

### ❌ DSA Mistakes

| Mistake | Fix |
|---|---|
| Looking at the solution after 5 minutes | Force yourself to attempt for 25–30 minutes minimum |
| Solving only easy problems | 70% of interview problems are medium — practice medium daily |
| Not reviewing wrong answers | Maintain an error log, re-solve wrong problems after 3 days |
| Memorizing solutions instead of patterns | Ask: "What pattern does this problem use?" not "What's the code?" |
| Skipping DP because it's hard | DP is in every top company's interview — you cannot avoid it |
| No timed practice | If you can't solve in 25 min under pressure, you'll fail live interviews |
| Starting with LeetCode Hard problems | Destroys confidence — build pattern recognition first with Easy/Medium |

### ❌ Development Mistakes

| Mistake | Fix |
|---|---|
| Building projects without deploying | Every project must have a live URL before adding it to resume |
| No README in repositories | Write README before considering a project "done" |
| Using Create React App for everything | Use Vite — faster, modern, what companies use |
| Hardcoding API keys in code | Always use .env files, always add to .gitignore |
| No error handling in APIs | 500 errors with no message = instant rejection in code reviews |
| Learning too many frameworks | Go deep on React + Node.js, not shallow on 10 stacks |
| No version control discipline | Commit often, use meaningful commit messages, use branches |

### ❌ Application Mistakes

| Mistake | Fix |
|---|---|
| Applying to only 5–10 companies | You need to apply to 100–200 to get 5–10 interviews |
| Using one generic resume for all jobs | Tailor the skills section to each job description keywords |
| Not following up | Follow up exactly once, 7 days after applying |
| Applying to only big companies (FAANG) | Target mid-size product companies (series A/B startups) — higher chance of getting in |
| Not asking for referrals | 60%+ of hires are through referrals — ask your network |
| Profile with zero activity | GitHub with no commits and LinkedIn with no posts screams "not serious" |
| Applying before being ready | Apply to practice companies first, save dream companies for when you're stronger |

### How to Increase Your Salary Chances

1. **Referrals:** A referral gets you into the hiring pipeline 10x faster and sometimes into a higher pay band.
2. **Competing Offers:** Always try to have 2–3 offers simultaneously. Use them for negotiation. "I have an offer from [Company X] at ₹Y, can you match or improve on that?"
3. **Strong Projects with Numbers:** Quantified impact on your resume commands higher initial offers.
4. **Negotiate Always:** Starting salary is negotiable. Saying "Can you do better?" costs nothing and works 60% of the time.
5. **ESOPs/RSUs:** For startups, ask about equity cliff and vesting schedule. Early-stage ESOPs can be worth a lot.
6. **Target the Right Companies:** Series B/C startups often pay as well as established product companies and hire more freshers.
